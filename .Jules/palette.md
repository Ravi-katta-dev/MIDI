## 2024-07-15 - Dynamic ARIA labels for Icon-Only Buttons
**Learning:** Found that many icon-only buttons (like Play, Pause, Zoom) were created dynamically via `DomHelper.createGlyphiconButton` but lacked accessible labels and tooltips, making them completely inaccessible for screen readers. Some buttons even had incorrect IDs from copy-pasting (e.g. speed down had the id speedUp).
**Action:** By modifying the centralized `createGlyphiconButton` utility, I was able to automatically derive a human-readable string from the button's ID (converting camelCase to Sentence case and stripping "Button" suffix) to apply as both `aria-label` and `title`. This is a powerful pattern to fix widespread accessibility issues in vanilla JS apps with centralized DOM helpers. Always check underlying IDs for copy-paste errors when relying on them for generated labels.

## 2024-07-20 - HTMLFor property vs for attribute in JS generated DOM elements
**Learning:** Found that when dynamically generating DOM elements and their attributes through object assignment (as done in `DomHelper.createElement`), properties must follow JavaScript DOM API naming conventions, not standard HTML attribute names. Specifically, attempting to use the `for` attribute in an options object (e.g. `{ for: id }`) will not properly serialize to the DOM label's `for` attribute. Furthermore, using `label.setAttribute("for", id)` in combination with other properties led to unexpected behavior and orphaned labels. When wrapper elements share the exact same `id` as the input they wrap, screen readers completely fail to associate the label with the input.
**Action:** Always use the DOM property `htmlFor` (e.g. `{ htmlFor: id }`) when directly assigning properties to generated label elements. Always ensure wrapper container IDs are distinct from their nested input element IDs (e.g. by appending "Cont" or "Container") to preserve valid HTML and ensure accessibility labels resolve correctly.
