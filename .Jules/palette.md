## 2024-07-15 - Dynamic ARIA labels for Icon-Only Buttons
**Learning:** Found that many icon-only buttons (like Play, Pause, Zoom) were created dynamically via `DomHelper.createGlyphiconButton` but lacked accessible labels and tooltips, making them completely inaccessible for screen readers. Some buttons even had incorrect IDs from copy-pasting (e.g. speed down had the id speedUp).
**Action:** By modifying the centralized `createGlyphiconButton` utility, I was able to automatically derive a human-readable string from the button's ID (converting camelCase to Sentence case and stripping "Button" suffix) to apply as both `aria-label` and `title`. This is a powerful pattern to fix widespread accessibility issues in vanilla JS apps with centralized DOM helpers. Always check underlying IDs for copy-paste errors when relying on them for generated labels.

## 2024-07-20 - HTML attributes vs JavaScript DOM properties for accessibility
**Learning:** Found that when constructing DOM elements dynamically in `js/ui/DomHelper.js`, the configuration objects map directly to DOM properties, not HTML attributes. Specifically, trying to set `{ for: id }` to link a `<label>` to an `<input>` fails because the correct DOM property is `htmlFor`. This resulted in broken label associations across checkboxes, sliders, and select dropdowns, breaking screen reader functionality and reducing clickable areas. Furthermore, some input elements were missing the `id` property entirely.
**Action:** Modified `DomHelper.js` to correctly use `{ htmlFor: id }` and ensure associated form elements (like checkboxes and selects) explicitly have an `id` that matches the label. Always remember to use JavaScript property names (like `htmlFor`, `className`) instead of HTML attributes when setting properties directly on DOM element objects.
