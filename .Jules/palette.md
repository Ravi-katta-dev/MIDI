## 2024-07-15 - Dynamic ARIA labels for Icon-Only Buttons
**Learning:** Found that many icon-only buttons (like Play, Pause, Zoom) were created dynamically via `DomHelper.createGlyphiconButton` but lacked accessible labels and tooltips, making them completely inaccessible for screen readers. Some buttons even had incorrect IDs from copy-pasting (e.g. speed down had the id speedUp).
**Action:** By modifying the centralized `createGlyphiconButton` utility, I was able to automatically derive a human-readable string from the button's ID (converting camelCase to Sentence case and stripping "Button" suffix) to apply as both `aria-label` and `title`. This is a powerful pattern to fix widespread accessibility issues in vanilla JS apps with centralized DOM helpers. Always check underlying IDs for copy-paste errors when relying on them for generated labels.

## 2026-07-22 - DOM Accessibility Attribute Mapping
**Learning:** Found that when dynamically generating DOM elements using a centralized helper (like `DomHelper.createElement` in vanilla JS), JavaScript object keys assigned to the element's properties map directly to DOM properties, not HTML attributes. This means using `for` to try to set the `for` attribute on a label will silently fail; it must be `htmlFor`. Also discovered ID collisions (e.g., passing the same string to both a container `id` and an input `id`) breaking label-to-input association.
**Action:** When updating vanilla JS DOM helpers for accessibility, manually verify that object keys map to valid DOM property names (e.g., `htmlFor`, `className`, `tabIndex`), and explicitly append suffixes like 'Cont' or 'Container' to wrapper IDs to ensure form inputs receive globally unique IDs that labels can successfully target via `htmlFor`.
