## 2024-07-15 - Dynamic ARIA labels for Icon-Only Buttons
**Learning:** Found that many icon-only buttons (like Play, Pause, Zoom) were created dynamically via `DomHelper.createGlyphiconButton` but lacked accessible labels and tooltips, making them completely inaccessible for screen readers. Some buttons even had incorrect IDs from copy-pasting (e.g. speed down had the id speedUp).
**Action:** By modifying the centralized `createGlyphiconButton` utility, I was able to automatically derive a human-readable string from the button's ID (converting camelCase to Sentence case and stripping "Button" suffix) to apply as both `aria-label` and `title`. This is a powerful pattern to fix widespread accessibility issues in vanilla JS apps with centralized DOM helpers. Always check underlying IDs for copy-paste errors when relying on them for generated labels.

## 2026-07-21 - Accessible Form Inputs and Elements in Vanilla JS
**Learning:** When generating complex form controls (like custom sliders, checkboxes, select boxes, and hidden file inputs wrapped in labels) dynamically via a utility like `DomHelper.js`, native HTML accessibility patterns often break because properties like `id` and `for` (or `htmlFor` in JS) must be explicitly managed and linked. Furthermore, interactive custom wrappers like `label` elements acting as buttons need `tabIndex`, `role="button"`, and explicit `keydown` handlers for Enter/Space to ensure full keyboard navigation and screen reader support.
**Action:** When building vanilla JS component systems, ensure `htmlFor` on labels correctly matches the `id` of the actual input element (not the wrapper), explicitly add keyboard event listeners to non-native button wrappers, and test keyboard navigability for custom input elements.
